# プロジェクト概要ドキュメント

## プロジェクト名
**TaskFlow** - チーム向けプロジェクト・タスク管理システム

---

## 1. 何を作るか（What）

### システム概要
中小規模のチーム（5-50人）向けのシンプルで使いやすいプロジェクト・タスク管理システム。
JIRA、Trello、Asanaのエッセンスを取り入れつつ、小規模チームに特化したシンプルさを重視。

### 主要機能
- **プロジェクト管理**: 複数プロジェクトの作成・管理
- **タスク管理**: タスクの作成、割り当て、ステータス管理
- **コラボレーション**: タスクへのコメント、メンション
- **可視化**: プロジェクト進捗ダッシュボード
- **通知**: タスク割り当て時の通知
- **検索・フィルタ**: タスクの高速検索

### 技術スタック
- **バックエンド**: Go 1.24 + Gin framework
- **データベース**: PostgreSQL
- **認証**: JWT
- **API**: RESTful API + OpenAPI 3.0
- **インフラ**: Docker + Docker Compose

---

## 2. なぜ作るか（Why）

### 解決する課題

#### 課題1: 既存ツールの複雑さ
- **現状**: JIRA、Asanaなどは機能が豊富だが、小規模チームには過剰
- **問題**: 学習コストが高い、セットアップに時間がかかる
- **解決**: 必要最低限の機能に絞り、即座に使い始められる

#### 課題2: 価格の壁
- **現状**: 多くのSaaSは従量課金（ユーザー数×月額）
- **問題**: スタートアップや小規模チームには負担
- **解決**: セルフホスト可能なオープンソース

#### 課題3: カスタマイズの難しさ
- **現状**: SaaSツールは自社のワークフローに合わせにくい
- **問題**: 運用ルールをツールに合わせる必要がある
- **解決**: オープンソースで自由にカスタマイズ可能

### 学習目的（このプロジェクトを通じて習得すること）
1. **実務的な開発プロセス**: 企画→設計→実装→機能追加
2. **Goでの本格的なWeb開発**: フレームワーク、DB、認証の統合
3. **アーキテクチャ設計**: レイヤードアーキテクチャの実践
4. **API設計**: RESTful API設計の実務レベル習得
5. **運用設計**: ログ、監視、障害対応の実装

---

## 3. 誰が使うか（Who）

### ターゲットユーザー

#### プライマリユーザー: 開発チーム
- **規模**: 5-20人程度
- **特徴**:
  - アジャイル開発を実践
  - スプリント単位での作業管理が必要
  - リモートワーク中心
- **利用シーン**:
  - 日次スタンドアップでタスク確認
  - スプリント計画でタスク割り当て
  - コードレビュー時の関連タスク参照

#### セカンダリユーザー: プロジェクトマネージャー
- **役割**: プロジェクト全体の進捗管理
- **ニーズ**:
  - プロジェクト進捗の可視化
  - ボトルネックの早期発見
  - レポート作成

#### セカンダリユーザー: ステークホルダー
- **役割**: プロジェクト状況の確認（読み取り専用）
- **ニーズ**:
  - プロジェクト進捗の把握
  - マイルストーン達成状況の確認

## 4. 利用シナリオ（User Stories）

### シナリオ1: 新規プロジェクト立ち上げ
```
背景: 新しいプロダクト開発が始まる
流れ:
1. PMがプロジェクト「Web API v2」を作成
2. 初期タスクを洗い出してTaskFlowに登録
3. 各タスクをチームメンバーに割り当て
4. メンバーはダッシュボードで自分のタスクを確認
```

### シナリオ2: 日次スタンドアップ
```
背景: チームの朝会
流れ:
1. 各メンバーが自分のタスクボードを開く
2. 「昨日完了したタスク」をステータス更新
3. 「今日取り組むタスク」を確認
4. ブロッカーがあればコメントで共有
```

### シナリオ3: スプリント計画
```
背景: 2週間スプリントの開始
流れ:
1. PMがバックログから優先タスクを選定
2. タスクを次スプリントに割り当て
3. 各タスクに担当者と期限を設定
4. チームメンバーが自分のタスクを確認して開始
```

### シナリオ4: タスク進行中
```
背景: 実装作業中に疑問が発生
流れ:
1. 開発者がタスクにコメントを追加
2. 担当者がメンション通知を受け取る
3. コメントで議論・解決
4. タスクを「完了」に変更
```

## 5. 非機能要件

### パフォーマンス
- API応答時間: 95パーセンタイルで300ms以下
- 同時接続: 50ユーザーまで対応
- タスク検索: 10,000件でも1秒以内

### セキュリティ
- 認証: JWT + パスワードハッシュ化（bcrypt）
- 認可: RBAC（Role-Based Access Control）
- 入力検証: すべてのエンドポイントでバリデーション

### 可用性
- 稼働率: 99%（月間ダウンタイム7時間以内）
- バックアップ: 日次自動バックアップ
- リカバリ: 1時間以内に復旧可能

### 運用性
- ログ: 構造化ログ（JSON形式）
- 監視: ヘルスチェックエンドポイント
- デプロイ: Docker Composeでワンコマンド起動

## 7. プロジェクト制約

### スコープ
- **Phase 1（Day61-74）**: MVP（最小限の機能）
  - プロジェクト作成・一覧
  - タスクCRUD
  - ユーザー認証
  - 基本的なステータス管理
- **Phase 2（Day75以降）**: 機能拡張
  - コメント機能
  - 通知機能
  - ダッシュボード
  - 検索・フィルタ

### 実装しない機能（今回は対象外）
- リアルタイム更新（WebSocket）→ Phase 3で検討
- ファイル添付 → Phase 3で検討
- カレンダー表示 → Phase 3で検討
- Slack/GitHub連携 → Phase 3で検討

### 技術的制約
- フロントエンドは作らない（APIのみ）
- 認証はJWTのみ（OAuth2は今回対象外）
- データベースはPostgreSQLのみ
